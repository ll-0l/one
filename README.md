# 업무 자동화 프롬프트 패키지

## 0. 과제 개요

### 0-1. 과제명

업무용 메일 초안 작성 자동화를 위한 LLM 비교·선정 및 프롬프트 설계 보고서

### 0-2. 과제 목적

본 과제의 목적은 업무용 메일 초안 작성이라는 실제 업무 과업을 대상으로, 서로 다른 3종 이상의 LLM을 비교하고 가장 적합한 모델을 선정한 뒤, 재사용 가능한 업무 자동화 프롬프트 패키지를 설계하는 것이다.

이를 위해 동일한 업무 과업 프롬프트를 여러 LLM에 입력하고, 모델별 결과를 정확성, 한국어 자연스러움, 형식 준수, 톤 조절, 환각 방지, 수정 대응력 등의 기준으로 평가한다. 이후 가장 적합한 모델을 선정하고, 선택한 과업에 맞는 페르소나, 시스템 프롬프트, 유저 입력 템플릿, Few-shot 예시, 환각 검증 전략, 실행 로그를 포함한 최종 프롬프트 패키지를 구성한다.

본 보고서는 LLM을 단순히 감각적으로 선택하는 것이 아니라, 업무 목적에 맞는 평가 기준과 근거를 바탕으로 비교·검증·개선하는 과정을 재현 가능하게 정리하는 데 초점을 둔다.

### 0-3. 선택한 업무 자동화 과업

본 과제에서 선택한 업무 자동화 과업은 **업무용 메일 초안 작성**이다.

사용자가 메일 작성 목적, 수신자, 상황, 필수 포함 항목, 원하는 톤, 금지 표현 등을 입력하면 LLM이 이를 바탕으로 정중하고 자연스러운 업무용 메일 초안을 작성하도록 설계한다.

## 1. LLM 모델 비교·선정 보고서

### 1-1. 비교 목적

본 비교의 목적은 “업무용 메일 초안 작성” 과업에 적합한 LLM을 선정하는 것이다. 단순히 문장이 자연스러운 모델을 고르는 것이 아니라, 사용자가 제공한 조건을 정확히 반영하는지, 없는 사실을 임의로 생성하지 않는지, 업무용 메일 형식을 안정적으로 지키는지, 수정 및 검증에 적합한 결과를 제공하는지를 종합적으로 평가한다.

특히 본 과업은 교육 운영 담당자에게 수강 일정 변경 가능 여부를 문의하는 메일을 작성하는 상황이므로, 다음 요소가 중요하다.

* 사용자가 제공한 과정명과 날짜를 정확히 반영하는가
* 일정 변경 가능 여부를 단정하지 않고 문의 형태로 표현하는가
* 담당자 이름, 이메일 주소, 정책, 규정 등을 임의로 생성하지 않는가
* 정중하지만 과도하게 사과하지 않는가
* 실제 발송 가능한 수준의 간결하고 자연스러운 업무 메일을 생성하는가
* 확인 필요 사항과 수정 제안을 별도로 구분하는가

### 1-5. 모델별 결과 요약

#### 1) Gemini 3.5 Flash 결과 요약

Gemini 3.5 Flash는 전체적으로 메일 형식을 잘 갖추었고, 제목과 본문이 실제 업무 상황에 어울릴 정도로 자연스러웠다. 과정명과 기존 신청 일정도 정확히 반영했으며, 일정 변경 가능 여부를 단정하지 않고 문의 형태로 작성했다.

다만 확인 필요 사항과 수정 제안에서 일부 추가 정보가 다소 넓게 제시되었다. 예를 들어 “잔여 좌석”, “교육 취소/변경 마감 기한”, “증빙 서류 제출 필요성” 등의 표현은 실제로 존재하는 정책이라고 단정하지는 않았지만, 사용자가 제공하지 않은 가능성을 비교적 적극적으로 제안했다. 또한 수정 제안에서 “예: 7월 24일”이라는 임의 날짜 예시를 제시한 점은 금지 조건인 “없는 정보 지어내기 금지” 관점에서 다소 주의가 필요하다.

종합하면 Gemini 3.5 Flash는 문장 품질과 형식은 우수하지만, 실무 검증 관점에서는 불필요한 예시나 가능성을 추가하는 경향이 있어 환각 방지 측면에서 감점 요인이 있었다.

#### 2) Claude Sonnet 5 결과 요약

Claude Sonnet 5는 간결하고 안정적인 업무 메일을 작성했다. 과정명, 기존 일정, 일정 변경 문의, 변경 가능한 날짜 안내 요청, 감사 표현이 모두 포함되어 있어 필수 조건을 충족했다. 또한 담당자 이름이나 이메일 주소를 임의로 만들지 않고 확인 필요 사항으로 분리한 점이 적절했다.

다만 결과 상단에 “제목”이 한 번 제시된 뒤 다시 “1. 메일 제목”이 반복되어 출력 형식이 약간 중복되었다. 본문은 자연스럽지만 마무리 서명이나 발신자 정보가 포함되지 않아 실제 발송 직전에는 사용자가 추가 정보를 보완해야 한다. 또한 “변경 사유를 조금 더 구체적으로 밝히면 좋다”는 수정 제안은 실무적으로 가능하지만, 개인정보 최소화 관점에서는 사용자의 선택 사항임을 명확히 구분해야 한다.

종합하면 Claude Sonnet 5는 정중함과 간결함의 균형이 좋고 환각 위험도 낮지만, 출력 구조의 중복과 세부 형식 안정성 면에서는 약간의 개선 여지가 있었다.

#### 3) GPT-5.5 Thinking 결과 요약

GPT-5.5 Thinking은 가장 간결하고 조건 준수 중심의 결과를 생성했다. 과정명, 기존 신청 일정, 일정 변경 가능 여부 문의, 변경 가능한 일정 안내 요청, 감사 표현을 모두 포함했으며, 과도한 사과 표현도 사용하지 않았다. 또한 담당자 이름, 이메일 주소, 구체적인 변경 가능 날짜, 정책 내용을 임의로 생성하지 않았다.

확인 필요 사항에서도 “실제로 변경 가능한 회차가 있는지 확인해야 한다”, “본인 확인이 필요한 경우 이름이나 신청 번호를 직접 추가하는 것이 좋다”, “변경 신청 기한이 있을 수 있으므로 가능한 빨리 발송하는 것이 좋다”처럼 불확실한 정보를 단정하지 않고 확인 대상으로 분리했다. 이는 본 과제에서 요구하는 환각 방지 기준에 잘 부합한다.

다만 메일 제목이 “수강 일정 변경 가능 여부 문의”로 다소 일반적이며, Gemini나 Claude에 비해 제목의 구체성은 조금 낮다. 본문 역시 매우 간결해 실제 업무 메일로는 안정적이지만, 격식 있는 보고서용 평가 관점에서는 조금 더 풍부한 표현이 가능하다.

종합하면 GPT-5.5 Thinking은 문장 화려함보다는 조건 준수, 환각 방지, 간결성, 실무 안정성 측면에서 가장 적합한 결과를 보였다.

### 1-7. 점수 부여 근거

#### Gemini 3.5 Flash

Gemini 3.5 Flash는 제목, 본문, 확인 필요 사항, 수정 제안 형식을 모두 지켰고, 문장도 자연스럽다. 특히 제목이 구체적이고 본문 흐름도 실제 메일에 가깝다. 그러나 “잔여 좌석”, “교육 취소/변경 마감 기한”, “증빙 서류 제출 필요성”, “예: 7월 24일”과 같이 사용자가 제공하지 않은 정보를 수정 제안에 비교적 적극적으로 포함했다. 해당 내용이 확정적으로 쓰인 것은 아니지만, 본 과업의 핵심 조건인 “없는 정책이나 규정 지어내기 금지” 관점에서는 주의가 필요하다. 따라서 환각 방지 항목은 3점으로 평가하였다.

#### Claude Sonnet 5

Claude Sonnet 5는 필수 포함 항목을 충실히 반영했고, 정중하고 간결한 업무 메일 톤을 유지했다. 담당자 이름과 이메일 주소를 임의로 만들지 않고 확인 필요 사항으로 분리한 점도 적절하다. 다만 출력 첫 부분에 제목이 중복 제시되어 형식 준수 측면에서 약간의 감점이 있었다. 또한 수정 제안에서 변경 사유를 더 구체적으로 밝히는 방안을 제안했는데, 이는 실무적으로 유용할 수 있으나 개인정보 최소화 관점에서는 신중하게 다루어야 한다. 전체적으로 안정적이지만 최종 자동화 모델로 선정하기에는 형식의 일관성과 환각 방지 측면에서 GPT-5.5 Thinking보다 조금 낮게 평가하였다.

#### GPT-5.5 Thinking

GPT-5.5 Thinking은 세 모델 중 가장 조건 준수와 환각 방지 측면에서 우수했다. 사용자가 제공한 과정명과 날짜만 사용했고, 변경 가능한 날짜나 담당자 정보, 내부 규정 등을 임의로 생성하지 않았다. 확인 필요 사항도 단정적 표현이 아니라 사용자가 실제 발송 전에 점검해야 할 항목으로 제시했다. 메일 제목과 본문은 다소 간결하지만, 본 과업의 핵심 목표가 “안전하고 재사용 가능한 업무 메일 자동화”라는 점을 고려하면 가장 안정적인 결과로 판단된다.

### 1-9. 최종 선정 근거

GPT-5.5 Thinking을 최종 모델로 선정한 이유는 다음과 같다.

첫째, 본 과업에서 가장 중요한 기준인 **환각 방지** 측면에서 가장 안정적이었다. Gemini 3.5 Flash는 예시 날짜나 증빙 서류 가능성 등 추가 정보를 비교적 적극적으로 제안했고, Claude Sonnet 5도 일부 보완 제안을 포함했으나, GPT-5.5 Thinking은 사용자가 제공하지 않은 정보를 단정하지 않고 확인 필요 사항으로 분리했다.

둘째, **금지 사항 준수**가 가장 명확했다. 과도한 사과 표현을 사용하지 않았고, 담당자 이름이나 이메일 주소를 임의로 생성하지 않았으며, 교육 운영 기관의 정책이나 변경 가능 여부를 확정적으로 말하지 않았다. 이는 업무 자동화 시스템에서 중요한 안정성 요건에 부합한다.

셋째, **실무 활용성**이 높았다. 메일 본문은 간결하고 바로 사용할 수 있는 형태였으며, 확인 필요 사항에서는 실제 발송 전 점검해야 할 항목을 분리해 제시했다. 특히 “본인 확인이 필요한 경우 신청자명 또는 신청 번호를 직접 추가하는 것이 좋다”는 제안은 개인정보 직접 포함 금지 조건과 실무 필요성을 균형 있게 반영한 결과로 볼 수 있다.

넷째, 향후 프롬프트 자동화 시스템으로 확장할 때 GPT-5.5 Thinking의 출력 방식이 가장 적합하다. 업무용 메일 자동화는 창의적인 표현보다 조건 준수, 형식 안정성, 누락 방지, 불확실성 표시가 중요하다. GPT-5.5 Thinking은 이 기준에서 가장 높은 점수를 받았으므로 최종 시스템 설계에 사용할 대표 모델로 선정한다.

### 1-10. 비교 결과 요약

| 항목                      | 최종 판단                             |
| ----------------------- | --------------------------------- |
| 문장 자연스러움이 가장 좋은 모델      | Gemini 3.5 Flash, Claude Sonnet 5 |

따라서 본 과제의 최종 업무 자동화 프롬프트 설계는 GPT-5.5 Thinking을 기준 모델로 삼아 진행한다.


2. **시스템 설계 문서**

## 1. 업무 과업 정의

### 1-1. 업무 과업명

업무용 메일 초안 작성 자동화

### 1-2. 문제 정의

업무용 메일은 수신자, 목적, 상황, 톤에 따라 적절한 표현이 달라진다. 특히 요청, 문의, 일정 변경, 사과, 자료 요청 등과 같은 상황에서는 지나치게 가볍거나 과도하게 비굴한 표현을 피하면서도 정중하고 명확한 문장을 구성해야 한다.

그러나 메일 작성 경험이 부족한 사용자는 다음과 같은 어려움을 겪을 수 있다.

* 어떤 제목을 붙여야 할지 판단하기 어렵다.
* 수신자에게 어느 정도로 정중하게 표현해야 하는지 모호하다.
* 요청 사항을 간결하게 전달하지 못하고 문장이 장황해진다.
* 날짜, 금액, 정책, 담당자명 등 확인되지 않은 정보를 임의로 포함할 위험이 있다.
* 메일의 목적과 필수 포함 항목이 누락될 수 있다.
* 조건이 중간에 변경되었을 때 기존 맥락을 유지하며 수정하기 어렵다.

따라서 본 과업은 사용자가 제공한 정보를 바탕으로 업무 상황에 적합한 메일 초안을 생성하되, 검증되지 않은 정보는 임의로 만들지 않고 확인 필요 사항으로 분리하는 것을 목표로 한다.

### 1-3. 타겟 사용자

본 시스템의 타겟 사용자는 업무용 메일을 빠르고 정확하게 작성해야 하는 사용자이다. 구체적인 대상은 다음과 같다.

* 업무용 이메일 작성 경험이 부족한 신입사원
* 교육 프로그램, 채용, 고객 응대, 일정 조율 등에서 메일을 자주 작성해야 하는 실무자
* 정중한 표현과 간결한 전달 방식을 동시에 맞추고 싶은 사용자
* 메일 초안을 작성한 뒤 톤, 형식, 누락 사항을 점검받고 싶은 사용자
* 개인정보나 확인되지 않은 정보를 포함하지 않도록 주의해야 하는 사용자

### 1-4. 입력 데이터 형태

사용자는 자연어 또는 템플릿 형태로 메일 작성에 필요한 정보를 입력한다.

주요 입력 데이터는 다음과 같다.

| 입력 항목      | 설명                            |
| ---------- | ----------------------------- |
| 메일 목적      | 문의, 요청, 사과, 일정 변경, 자료 요청 등    |
| 수신자/대상     | 담당자, 교수, 회사 인사팀, 고객, 협력사 등    |
| 상황 설명      | 메일을 보내야 하는 배경                 |
| 필수 포함 항목   | 반드시 본문에 들어가야 하는 정보            |
| 원하는 톤      | 정중하게, 간결하게, 단호하게, 부드럽게 등      |
| 금지 표현      | 과도한 사과, 추측성 표현, 특정 단어 등       |
| 개인정보 처리 방식 | 실명, 전화번호, 주소 등 민감 정보 포함 여부    |
| 확인 필요 정보   | 날짜, 정책, 금액, 담당자명 등 검증이 필요한 정보 |

### 1-5. 출력 규격

LLM은 사용자의 입력을 바탕으로 다음 형식에 맞춰 결과를 출력한다.

```md
## 메일 제목
...

## 메일 본문
...

## 확인 필요 사항
- ...

## 수정 제안
- ...
```

출력 결과는 다음 원칙을 따른다.

* 메일 제목은 목적이 드러나도록 간결하게 작성한다.
* 본문은 인사말, 상황 설명, 요청 사항, 마무리 인사 순서로 구성한다.
* 사용자가 제공하지 않은 날짜, 금액, 정책, 담당자명, 이메일 주소는 임의로 생성하지 않는다.
* 확인이 필요한 정보는 본문에 단정적으로 넣지 않고 별도 항목으로 분리한다.
* 사용자가 지정한 톤과 금지 표현을 반영한다.
* 개인정보는 사용자가 명시적으로 제공하고 포함을 허용한 경우에만 사용한다.

### 1-6. 재사용 가능한 입력 템플릿

아래 템플릿은 사용자가 반복적으로 업무용 메일 초안을 요청할 때 사용할 수 있다.

```md
[메일 목적]
예: 일정 변경 문의 / 자료 요청 / 사과 메일 / 지원 문의 / 회신 요청

[수신자/대상]
예: 교육 운영 담당자 / 회사 인사팀 / 협력사 담당자 / 교수님 / 고객

[상황 설명]
예: 기존에 신청한 교육 일정 참석이 어려워져 변경 가능 여부를 문의해야 함

[필수 포함 항목]
- 과정명:
- 기존 일정:
- 변경 희망 일정:
- 요청 사항:
- 감사 표현:

[원하는 톤]
예: 정중하고 간결하게 / 부드럽지만 명확하게 / 격식 있게 / 단호하게

[금지 표현]
예: 과도한 사과 금지 / 추측성 표현 금지 / 없는 정책 언급 금지

[개인정보 처리 방식]
예: 실명 제외 / 이름만 포함 / 연락처 제외 / 개인정보 직접 포함 금지

[확인 필요한 정보]
예: 정확한 일정, 담당자명, 환불 규정, 첨부파일 여부 등
```

### 1-7. 이번 과업의 테스트용 동일 입력 프롬프트

아래 프롬프트는 Gemini 3.5 Flash, Claude Sonnet 5, GPT-5.5 Thinking에 동일하게 입력하여 모델별 결과를 비교하는 데 사용한다.

[필수 포함 항목]
