# 업무 자동화 프롬프트 패키지

## 0. 과제 개요

### 0-1. 과제명

업무용 메일 초안 작성 자동화를 위한 LLM 비교·선정 및 프롬프트 설계 보고서

### 0-2. 과제 목적

본 과제의 목적은 **교육 프로그램 운영 회사의 업무용 메일 초안 작성**이라는 실제 업무 과업을 대상으로, 서로 다른 3종 이상의 LLM을 비교하고 가장 적합한 모델을 선정한 뒤, 재사용 가능한 업무 자동화 프롬프트 패키지를 설계하는 것이다.

이를 위해 동일한 업무 과업 프롬프트를 여러 LLM에 입력하고, 모델별 결과를 업무 맥락 이해도, 정확성, 한국어 자연스러움, 형식 준수, 환각 방지, 실무 활용성 등의 기준으로 평가한다. 이후 가장 적합한 모델을 선정하고, 선택한 과업에 맞는 페르소나, 시스템 프롬프트, 유저 입력 템플릿, Few-shot 예시, 환각 검증 전략, 실행 로그를 포함한 최종 프롬프트 패키지를 구성한다.

본 보고서는 LLM을 단순히 감각적으로 선택하는 것이 아니라, 업무 목적에 맞는 평가 기준과 근거를 바탕으로 비교·검증·개선하는 과정을 재현 가능하게 정리하는 데 초점을 둔다.

### 0-3. 선택한 업무 자동화 과업

본 과제에서 선택한 업무 자동화 과업은 **교육 프로그램 운영 회사의 업무용 메일 초안 작성 자동화**이다.

교육 운영팀 신입사원 또는 사회초년생이 메일 작성 목적, 작성자 역할, 수신자, 상황, 과정명, 관련 일정, 필수 포함 항목, 원하는 톤, 금지 사항 등을 입력하면 LLM이 이를 바탕으로 정중하고 간결한 업무용 메일 초안을 작성하도록 설계한다.

### 0-4. 비교 대상 모델

본 과제에서는 다음 3종의 LLM을 비교 대상으로 선정한다.

| 구분   | 모델명              | 사용 환경 | 요금제 | 사용 날짜      |
| ---- | ---------------- | ----- | --- | ---------- |
| 모델 A | Gemini 3.5 Flash | 웹     | 무료  | 2026-07-06 |
| 모델 B | Claude Sonnet 5  | 웹     | 무료  | 2026-07-06 |
| 모델 C | GPT-5.5 Thinking | 웹     | 유료  | 2026-07-06 |

무료 버전으로 사용한 Gemini 3.5 Flash와 Claude Sonnet 5는 응답 횟수, 모델 선택, 세부 설정값 조정 등에 제한이 있을 수 있다. GPT-5.5 Thinking은 유료 환경에서 사용하므로 상대적으로 안정적인 사용이 가능하나, 웹 환경에서는 temperature, top_p, max_tokens 등 세부 생성 설정값을 사용자가 직접 조정하기 어렵다는 한계가 있다.

## 1. 업무 과업 정의 및 진행 사항

### 1-1. 업무 과업명

업무용 메일 초안 작성 자동화

### 1-2. 업무 과업 개요

본 과제에서 선택한 업무 자동화 과업은 **교육 프로그램 운영 회사의 업무용 메일 초안 작성 자동화**이다.

본 시스템은 교육 운영팀 신입사원 또는 사회초년생이 수강생, 신청자, 외부 담당자에게 보내는 업무용 메일을 빠르고 정확하게 작성할 수 있도록 돕는 것을 목표로 한다. 사용자가 메일 목적, 작성자 역할, 수신자, 상황, 과정명, 관련 일정, 필수 포함 항목, 원하는 톤, 금지 사항을 입력하면 LLM이 이를 바탕으로 정중하고 간결한 업무용 메일 초안을 작성한다.

특히 교육 운영 업무에서는 수강 일정 변경, 교육 안내, 자료 요청, 신청 정보 확인, 수료 안내 등 반복적인 메일 작성이 자주 발생한다. 이 과정에서 확인되지 않은 일정, 내부 정책, 담당자 정보, 이메일 주소, 환불 규정 등을 임의로 작성하면 잘못된 안내가 될 수 있으므로, 본 과업은 **정확성, 형식 준수, 개인정보 보호, 환각 방지**를 중요한 기준으로 삼는다.

### 1-3. 문제 정의

교육 프로그램을 운영하는 회사의 신입사원이나 사회초년생은 업무용 메일 작성 경험이 충분하지 않을 수 있다. 특히 수강생 문의에 회신하거나 외부 담당자에게 자료를 요청하는 상황에서는 정중한 표현, 간결한 문장, 정확한 정보 전달이 모두 필요하다.

그러나 실제 업무에서는 다음과 같은 어려움이 발생할 수 있다.

* 메일 제목을 어떻게 작성해야 할지 판단하기 어렵다.
* 수신자에게 어느 정도로 정중하게 표현해야 하는지 모호하다.
* 상황 설명과 요청 사항을 간결하게 정리하지 못해 문장이 장황해질 수 있다.
* 과정명, 일정, 신청 정보 등 필수 정보가 누락될 수 있다.
* 변경 가능 여부, 내부 규정, 환불 기준 등 확인되지 않은 정보를 단정적으로 안내할 위험이 있다.
* 담당자명, 이메일 주소, 신청자 개인정보 등을 임의로 포함할 수 있다.
* 중간에 조건이 변경되었을 때 기존 맥락을 유지하며 수정하기 어렵다.

따라서 본 과업은 교육 운영 회사의 실무자가 핵심 정보를 입력하면 LLM이 이를 바탕으로 정중하고 간결한 업무용 메일 초안을 작성하고, 검증이 필요한 정보는 별도 항목으로 분리하도록 설계하는 것을 목표로 한다.

### 1-4. 타겟 사용자

본 시스템의 타겟 사용자는 **교육 프로그램 운영 회사에서 업무용 메일을 작성해야 하는 신입사원 및 사회초년생**이다.

구체적인 사용자는 다음과 같다.

* 교육 프로그램 운영팀에 처음 배치된 신입사원
* 수강생 문의, 일정 변경, 교육 안내 메일을 작성해야 하는 사회초년생
* 업무용 이메일 작성 경험이 부족한 교육 운영 실무자
* 정중하지만 간결한 메일 문장을 빠르게 작성해야 하는 담당자
* 내부 정책, 일정, 신청 정보 등 확인이 필요한 내용을 임의로 작성하지 않도록 점검이 필요한 사용자

본 타겟 사용자는 교육 운영 업무를 수행하는 입장이므로, 메일 작성 시 단순한 친절함보다 정확성, 형식 준수, 개인정보 보호, 확인 필요 사항 분리가 중요하다.

### 1-5. 업무 상황 재정의

본 과업의 업무 상황은 다음과 같이 정의한다.

본 시스템은 “개인 수강생이 교육 운영 담당자에게 문의하는 메일”을 작성하는 도구가 아니라, **교육 운영 회사의 실무자가 수강생 또는 관계자에게 발송하는 업무 메일을 자동화하는 도구**로 설계한다.

### 1-7. 출력 규격

LLM은 사용자의 입력을 바탕으로 다음 형식에 맞춰 결과를 출력한다.

```md
## 1. 메일 제목
...

## 2. 메일 본문
...

## 3. 확인 필요 사항
- ...

## 4. 수정 제안
- ...
```

출력 결과는 다음 원칙을 따른다.

* 메일 제목은 목적이 드러나도록 간결하게 작성한다.
* 본문은 인사말, 상황 설명, 요청 또는 안내 사항, 마무리 인사 순서로 구성한다.
* 사용자가 제공하지 않은 날짜, 금액, 정책, 담당자명, 이메일 주소는 임의로 생성하지 않는다.
* 확인이 필요한 정보는 본문에 단정적으로 넣지 않고 별도 항목으로 분리한다.
* 사용자가 지정한 톤과 금지 사항을 반영한다.
* 개인정보는 사용자가 명시적으로 제공하고 포함을 허용한 경우에만 사용한다.

### 1-8. 재사용 가능한 입력 템플릿

```md
[메일 목적]
예: 일정 변경 안내 / 수강 일정 변경 가능 여부 회신 / 교육 안내 / 자료 요청 / 신청 정보 확인 / 회신 요청

[작성자 역할]
예: 교육 운영팀 신입사원 / 교육 프로그램 담당자 / 운영 지원 담당자

[수신자/대상]
예: 수강생 / 교육 신청자 / 외부 교육 담당자 / 협력사 담당자

예: 수강생이 AI 업무 자동화 실습 과정의 기존 신청 일정 변경 가능 여부를 문의함

[과정명]

[필수 포함 항목]
- 문의 접수에 대한 안내
- 일정 변경 가능 여부 확인 필요
- 변경 가능한 날짜가 있는 경우 안내 예정
- 추가 확인이 필요한 정보 요청
- 답변 또는 확인에 대한 감사 표현

[원하는 톤]
예: 정중하고 간결한 업무 메일 / 격식 있지만 딱딱하지 않게 / 친절하고 명확하게

[금지 사항]
예: 없는 정책이나 규정 지어내기 금지 / 담당자 이름이나 이메일 주소 임의 생성 금지 / 과도한 사과 표현 금지 / 개인정보 직접 포함 금지

[개인정보 처리 방식]
예: 실명 제외 / 신청자명은 사용자가 직접 입력 / 연락처 직접 포함 금지 / 접수번호는 확인 필요로 표시

[출력 형식]
1. 메일 제목
2. 메일 본문
3. 확인 필요 사항
4. 수정 제안
```

### 1-9. 테스트용 동일 입력 프롬프트

아래 프롬프트는 Gemini 3.5 Flash, Claude Sonnet 5, GPT-5.5 Thinking에 동일하게 입력하여 모델별 결과를 비교하는 데 사용하였다.


## 2. LLM 모델 비교·선정 보고서

### 2-1. 비교 목적

본 비교의 목적은 **교육 프로그램 운영 회사에서 신입사원 및 사회초년생이 반복적으로 작성해야 하는 업무용 메일 초안 작성 과업에 가장 적합한 LLM을 선정**하는 것이다.

본 과업은 단순히 자연스러운 문장을 생성하는 것이 아니라, 교육 운영 업무에서 자주 발생하는 수강 일정 변경 문의, 교육 일정 안내, 자료 요청, 신청 정보 확인 등의 상황에 맞춰 정중하고 정확한 메일 초안을 작성하는 것을 목표로 한다. 특히 교육 운영 메일에는 과정명, 일정, 신청 정보, 변경 가능 여부, 내부 정책, 담당자 확인 사항 등 사실 확인이 필요한 정보가 포함될 수 있으므로, LLM이 없는 정보를 임의로 생성하지 않는지가 중요하다.

따라서 본 비교에서는 각 모델이 다음 기준을 얼마나 충족하는지 평가한다.

* 교육 운영 업무 맥락을 정확히 이해하는가
* 신입사원 및 사회초년생이 바로 활용할 수 있는 업무용 메일 형식을 제공하는가
* 수강생 또는 신청자에게 보내는 회신 메일의 톤을 정중하고 간결하게 유지하는가
* 사용자가 제공한 과정명, 일정, 목적, 수신자 정보를 정확히 반영하는가
* 일정 변경 가능 여부, 내부 규정, 담당자 정보 등을 임의로 단정하지 않는가
* 확인이 필요한 정보와 실제 메일 본문을 구분하여 제시하는가
* 출력 형식인 메일 제목, 메일 본문, 확인 필요 사항, 수정 제안을 안정적으로 준수하는가

### 2-3. 비교에 사용한 동일 입력

### 2-5. 모델별 결과 요약

#### 1) Gemini 3.5 Flash 결과 요약

Gemini 3.5 Flash는 “교육운영팀 담당자”의 입장에서 수강생에게 회신하는 구조를 명확하게 반영했다. 메일 제목에 `[교육운영팀]`을 포함하여 발신 주체를 분명히 했고, 본문에서도 “교육운영팀 담당자입니다”, “일정 변경 요청을 정상적으로 접수하였습니다”와 같이 교육 운영 회사의 실무자가 작성하는 메일이라는 맥락을 잘 살렸다.

과정명인 `AI 업무 자동화 실습 과정`과 기존 신청 일정인 `2026년 7월 10일`도 정확하게 반영했다. 또한 일정 변경 가능 여부와 변경 가능한 대체 날짜가 아직 확인되지 않았다는 점을 고려하여 “확인하는 대로 다시 안내”하겠다고 표현한 점은 적절하다.

다만 본문에서 “다른 일정 잔여 좌석”이라는 표현을 사용했는데, 잔여 좌석 확인은 실제 교육 운영에서 필요한 요소일 수 있으나 입력에서 직접 제공된 정보는 아니다. 또한 수정 제안에서 “영업일 기준 1일 이내”와 같은 구체적 처리 시간을 예시로 제시했는데, 조건부 표현으로 제안되었더라도 실제 내부 기준이 없는 경우에는 주의가 필요하다. 따라서 문장 자연스러움과 업무 맥락 반영은 우수하지만, 환각 방지 측면에서는 약간의 감점 요인이 있다.

종합하면 Gemini 3.5 Flash는 가장 풍부하고 실무적인 느낌의 메일을 작성했으나, 확인되지 않은 운영 요소를 비교적 적극적으로 제안하는 경향이 있어 신입사원용 자동화 시스템에서는 추가적인 제약 문구가 필요하다.

#### 2) Claude Sonnet 5 결과 요약

Claude Sonnet 5는 정중하고 간결한 업무 메일을 안정적으로 작성했다. 수강생이 문의한 내용을 확인했다는 점, 기존 신청 일정이 2026년 7월 10일이라는 점, 일정 변경 가능 여부는 내부 확인 후 안내하겠다는 점을 명확하게 반영했다.

또한 변경 가능한 날짜가 있는 경우 확인되는 대로 추가 안내하겠다고 표현하여, 변경 가능 여부를 단정하지 않았다. 신청자 확인 정보가 필요할 수 있다는 점도 본문에 자연스럽게 포함했으며, 확인 필요 사항에는 대체 일정 존재 여부, 내부 절차 또는 규정, 신청자 확인 정보, 회신 담당자 정보 등을 구분해 제시했다.

다만 확인 필요 사항과 수정 제안에서 “회신 담당자명 및 연락처”, “예상 기간 1~2일 이내” 등 실제로는 내부 기준 확인이 필요한 요소를 제안했다. 이 역시 단정적으로 생성한 것은 아니지만, 신입사원이 그대로 사용할 경우 실제 회사 기준과 다를 수 있으므로 확인 필요 항목으로 다루어야 한다. 또한 본문은 안정적이지만, Gemini에 비해 발신 부서나 운영팀 맥락이 조금 덜 구체적으로 드러난다.

종합하면 Claude Sonnet 5는 정중함, 간결함, 안정성의 균형이 좋고, 과업 조건을 대부분 잘 준수했다. 다만 자동화 시스템의 최종 모델로 선정하기 위해서는 출력 형식의 세부 통제와 내부 기준 관련 표현을 조금 더 보수적으로 다룰 필요가 있다.

#### 3) GPT-5.5 Thinking 결과 요약

GPT-5.5 Thinking은 세 모델 중 조건 준수와 환각 방지 측면에서 가장 안정적인 결과를 보였다. 과정명, 기존 신청 일정, 일정 변경 문의 접수, 내부 확인 후 안내, 변경 가능한 날짜가 있는 경우 추가 안내, 신청자 정보 확인 가능성 등 필수 포함 항목을 모두 반영했다.

특히 일정 변경 가능 여부를 확정하지 않고 “내부 확인 후 안내드리겠습니다”라고 표현했으며, 변경 가능한 날짜도 임의로 만들지 않았다. 확인 필요 사항에서도 신청 정보, 신청 내역, 다른 운영 일정 존재 여부, 일정 변경 가능 여부, 안내 가능 범위를 분리하여 제시했다. 이는 신입사원이나 사회초년생이 실제 발송 전 무엇을 확인해야 하는지 파악하는 데 도움이 된다.

다만 본문이 매우 간결하여, 실제 고객 응대 메일로 사용할 경우 회사명, 교육운영팀 서명, 담당자명 등 실무 정보가 추가되면 더 완성도가 높아질 수 있다. 또한 “조금만 기다려 주시면 감사하겠습니다”라는 표현은 친절하지만, 격식 있는 직장 보고서용 톤에서는 “확인 후 안내드리겠습니다” 정도로 더 간결하게 조정할 수 있다.

종합하면 GPT-5.5 Thinking은 문장 표현의 풍부함보다는 정확성, 조건 준수, 환각 방지, 확인 필요 사항 분리에 강점을 보였다. 업무 자동화 시스템의 안정성을 기준으로 볼 때 최종 모델로 가장 적합하다.

### 2-7. 점수 부여 근거

#### Gemini 3.5 Flash

Gemini 3.5 Flash는 교육 운영팀이 수강생에게 회신하는 상황을 가장 구체적으로 표현했다. 제목에 교육운영팀을 명시하고, 본문에서도 문의 접수, 일정 확인, 추가 안내 예정 등 실무 흐름을 자연스럽게 구성했다. 문장도 정중하고 부드러워 실제 고객 응대 메일에 가깝다.

다만 “잔여 좌석”, “성함, 연락처 등”, “영업일 기준 1일 이내”와 같은 표현은 실제 교육 운영 상황에서 필요할 수 있으나, 입력에서 직접 제공된 정보는 아니다. 특히 처리 소요 시간은 내부 기준이 없는 상태에서 예시로라도 구체화하면 오해가 생길 수 있다. 따라서 업무 맥락과 문장 품질은 높지만, 환각 방지와 보수적 안내 측면에서는 일부 감점하였다.

#### Claude Sonnet 5

Claude Sonnet 5는 필수 조건을 충실히 반영하고, 정중하고 간결한 문장으로 안정적인 회신 메일을 작성했다. 변경 가능 여부를 단정하지 않고 내부 확인 후 안내하겠다고 표현했으며, 변경 가능한 날짜가 있는 경우 추가 안내하겠다는 조건부 표현도 적절했다.

확인 필요 사항에서는 대체 일정 존재 여부, 내부 절차 또는 규정, 신청자 확인 정보, 회신 담당자 정보 등을 제시하여 실무자가 발송 전 확인해야 할 항목을 잘 정리했다. 다만 발신 주체인 교육운영팀의 구체성이 Gemini보다 약하고, “1~2일 이내”와 같은 예상 기간 예시는 내부 기준이 있을 때만 사용할 수 있으므로 주의가 필요하다. 전체적으로 균형이 좋고 실무 활용성이 높은 결과로 평가하였다.

#### GPT-5.5 Thinking

GPT-5.5 Thinking은 본 과업의 핵심 기준인 정확성, 형식 준수, 환각 방지에서 가장 높은 평가를 받았다. 사용자가 제공한 과정명과 일정만 활용했고, 변경 가능한 날짜나 내부 정책을 임의로 생성하지 않았다. 또한 확인이 필요한 사항을 본문에 단정적으로 포함하지 않고 별도 항목으로 분리하여, 실제 발송 전 검토가 가능하도록 구성했다.

메일 본문은 다른 모델보다 간결하지만, 교육 운영 업무 자동화에서는 과도하게 풍부한 표현보다 정확하고 안전한 안내가 더 중요하다. 특히 신입사원 및 사회초년생이 사용할 시스템에서는 잘못된 일정, 내부 규정, 담당자 정보가 포함되지 않도록 통제하는 것이 중요하므로 GPT-5.5 Thinking을 가장 높게 평가하였다.

### 2-9. 최종 선정 근거

GPT-5.5 Thinking을 최종 모델로 선정한 이유는 다음과 같다.

첫째, 본 시스템의 타겟 사용자인 **교육 운영 회사의 신입사원 및 사회초년생**에게 가장 중요한 것은 잘못된 안내를 줄이는 것이다. GPT-5.5 Thinking은 변경 가능한 날짜, 내부 규정, 담당자 정보, 이메일 주소 등을 임의로 생성하지 않고 확인 필요 사항으로 분리하여, 실제 업무에서 발생할 수 있는 환각 위험을 가장 효과적으로 낮췄다.

둘째, **업무 메일 자동화에 필요한 형식 안정성**이 높았다. 메일 제목, 메일 본문, 확인 필요 사항, 수정 제안의 구조를 안정적으로 유지했으며, 사용자가 제공한 정보와 추가 확인이 필요한 정보를 구분했다. 이는 반복 사용 가능한 프롬프트 시스템을 설계하는 데 중요한 장점이다.

셋째, **금지 사항 준수 능력**이 가장 우수했다. 과도한 사과 표현을 사용하지 않았고, 일정 변경 가능 여부를 확정적으로 말하지 않았으며, 내부 정책이나 환불 규정처럼 근거가 필요한 내용을 지어내지 않았다. 따라서 교육 운영 실무자가 수강생에게 보내는 메일 초안 작성에 적합하다.

넷째, **실무 활용성**이 높다. 신입사원은 메일을 작성할 때 무엇을 본문에 넣고 무엇을 확인해야 하는지 구분하기 어려울 수 있다. GPT-5.5 Thinking은 본문과 확인 필요 사항을 분리하여, 사용자가 실제 발송 전에 일정, 신청자 정보, 내부 처리 기준 등을 점검할 수 있도록 도왔다.

다섯째, 재테스트 결과에서도 GPT-5.5 Thinking은 수정된 업무 상황을 안정적으로 반영했다. 기존의 개인 수강생 문의 메일 구조가 아니라 교육 운영팀의 회신 메일 구조로 바뀌었음에도, 핵심 조건을 유지하면서 과업 목적에 맞는 결과를 생성했다. 따라서 최종 시스템 설계의 기준 모델로 사용하기에 가장 적합하다고 판단하였다.

## 3. 시스템 설계 문서

### 3-1. 문제 정의

본 시스템 설계 문서는 앞서 정의한 업무 과업을 실제 프롬프트 시스템으로 구현하기 위한 설계 내용을 정리한 것이다. 시스템의 목적은 교육 프로그램을 운영하는 회사에서 신입사원 및 사회초년생이 반복적으로 작성해야 하는 업무용 메일을 효율적으로 작성하도록 돕는 것이다.

교육 운영 업무에서는 수강 신청, 일정 변경, 교육 안내, 자료 요청, 출석 확인, 수료 안내 등 다양한 상황에서 수강생 또는 외부 담당자에게 메일을 작성해야 한다. 그러나 업무 경험이 부족한 신입사원이나 사회초년생은 메일의 목적, 수신자, 상황, 필수 포함 항목, 표현 톤을 적절히 정리하는 데 어려움을 겪을 수 있다.

특히 교육 프로그램 관련 메일은 일정, 과정명, 신청 정보, 변경 가능 여부, 담당자 확인 사항 등 사실 확인이 필요한 정보가 자주 포함된다. 이때 확인되지 않은 일정, 내부 정책, 담당자명, 이메일 주소, 환불 규정 등을 임의로 작성하면 잘못된 안내가 될 수 있다.

따라서 본 과업은 교육 운영 회사의 실무자가 핵심 정보를 입력하면 LLM이 이를 바탕으로 정중하고 간결한 업무용 메일 초안을 작성하고, 검증이 필요한 정보는 별도 항목으로 분리하도록 설계하는 것을 목표로 한다.

본 시스템은 단순히 문장을 자연스럽게 만들어주는 도구가 아니라, 신입 실무자가 업무 메일을 작성할 때 발생할 수 있는 누락, 과도한 표현, 추측성 안내, 개인정보 노출, 환각 문제를 줄이는 업무 자동화 도구로 활용하는 것을 목적으로 한다.

### 3-2. 타겟 사용자

본 시스템의 타겟 사용자는 앞서 정의한 과업 기준과 동일하게 **교육 프로그램 운영 회사에서 업무용 메일을 작성해야 하는 신입사원 및 사회초년생**이다.

구체적인 사용자는 다음과 같다.

* 교육 프로그램 운영팀에 처음 배치된 신입사원
* 수강생 문의, 일정 변경, 교육 안내 메일을 작성해야 하는 사회초년생
* 업무용 이메일 작성 경험이 부족한 교육 운영 실무자
* 정중하지만 간결한 메일 문장을 빠르게 작성해야 하는 담당자
* 내부 정책, 일정, 신청 정보 등 확인이 필요한 내용을 임의로 작성하지 않도록 점검이 필요한 담당자

본 타겟 사용자는 교육 운영 업무를 수행하는 입장이므로, 메일 작성 시 단순한 친절함보다 정확성, 형식 준수, 개인정보 보호, 확인 필요 사항 분리가 중요하다.

### 3-5. 페르소나

#### 페르소나 이름

메일 코치 민서

#### 역할

교육 운영 업무에 특화된 비즈니스 메일 작성 코치

#### 전문 분야

* 교육 프로그램 안내 메일
* 수강 신청 및 일정 변경 관련 메일
* 수강생 문의 회신 메일
* 자료 요청 및 회신 요청 메일
* 신청 정보 확인 메일
* 정중한 업무용 커뮤니케이션 문장 작성

#### 말투

간결하고 정중한 직장 보고서식 표현을 사용한다. 지나치게 친근하거나 감정적인 표현은 피하고, 수신자가 바로 이해할 수 있도록 명확한 문장으로 작성한다.

#### 우선순위

1. 정확성
2. 개인정보 보호
3. 형식 준수
4. 자연스러운 한국어
5. 친절함

#### 금지 사항

* 사용자가 제공하지 않은 일정, 정책, 금액, 담당자명, 이메일 주소를 임의로 생성하지 않는다.
* 수강 일정 변경 가능 여부를 확인하지 않고 확정적으로 안내하지 않는다.
* 내부 규정, 환불 규정, 취소 규정을 근거 없이 단정하지 않는다.
* 개인정보를 사용자가 명시하지 않은 상태에서 직접 포함하지 않는다.
* 과도한 사과 표현이나 불필요하게 장황한 문장을 사용하지 않는다.
* 모호한 요청에 대해 필요한 확인 없이 임의로 상황을 만들어 작성하지 않는다.

### 3-6. 시스템 프롬프트 v1

```md
너는 업무용 메일을 작성하는 AI이다. 사용자의 요청을 바탕으로 정중하고 자연스러운 메일 초안을 작성하라.
```

#### v1의 한계

v1은 역할과 목표가 매우 단순하게 정의되어 있어, 사용자가 제공하지 않은 정보까지 AI가 임의로 보완할 가능성이 있다. 또한 교육 운영 업무라는 구체적인 맥락, 개인정보 보호, 확인 필요 사항 분리, 출력 형식 규칙이 부족하다. 따라서 실제 업무 자동화에 사용하기에는 환각 방지와 형식 안정성 측면에서 한계가 있다.

### 3-7. 시스템 프롬프트 v2

```md
너는 교육 운영 업무에 특화된 비즈니스 메일 작성 코치이다.

목표는 교육 프로그램 운영 회사의 신입사원 및 사회초년생이 수강생, 신청자, 외부 담당자에게 보낼 업무용 메일 초안을 정확하고 정중하게 작성하도록 돕는 것이다.

작업 절차:
1. 먼저 사용자의 입력에서 메일 목적, 작성자 역할, 수신자, 상황, 과정명, 관련 일정, 필수 포함 항목, 원하는 톤, 금지 사항을 파악한다.
2. 사용자가 제공한 정보만 바탕으로 메일을 작성한다.
3. 일정, 정책, 금액, 담당자명, 이메일 주소, 환불 규정, 변경 가능 여부 등 검증이 필요한 정보는 임의로 만들지 않는다.
4. 정보가 부족하지만 초안 작성이 가능한 경우에는 본문에서는 단정하지 않고, 별도의 [확인 필요 사항]에 표시한다.
5. 정보가 너무 부족해 메일 작성이 어려운 경우에는 최대 3개까지만 확인 질문을 한다.
6. 개인정보는 사용자가 명시적으로 제공하고 포함을 요청한 경우에만 사용한다. 다만 이름, 연락처, 이메일 주소 등 개인정보 항목은 기본적으로 본문에 직접 나열하지 않고, 필요한 경우 별도 안내 또는 확인 필요 사항으로 분리한다.
7. 과도한 사과 표현, 추측성 표현, 불필요하게 장황한 문장은 피한다.
8. 내부적으로 단계적으로 검토하되, 최종 답변에는 장문의 추론 과정을 노출하지 않는다.

출력 형식:
## 1. 메일 제목
- 목적이 드러나도록 간결하게 작성한다.

## 2. 메일 본문
- 인사말, 상황 설명, 요청 또는 안내 사항, 마무리 인사 순서로 작성한다.
- 정중하고 간결한 업무 메일 톤을 유지한다.

## 3. 확인 필요 사항
- 사용자가 발송 전 확인해야 할 일정, 정책, 신청자 정보, 담당자 정보 등을 정리한다.

## 4. 수정 제안
- 더 명확하게 만들기 위해 추가하면 좋은 정보나 문장 조정 방향을 제안한다.
```

### 3-9. 환각 검증 설계

* 회신 예상 시점, 처리 기한, 변경 가능 날짜는 내부 기준이 확인된 경우에만 작성한다.
* 미확정 정보는 본문에 단정적으로 넣지 않고 확인 필요 사항으로 분리한다.

본 과제에서는 교육 프로그램 운영 회사의 신입사원 및 사회초년생이 반복적으로 작성해야 하는 업무용 메일 초안 작성 과업을 대상으로, 3종의 LLM을 비교하고 최종 프롬프트 패키지를 설계하였다.

비교 대상 모델은 Gemini 3.5 Flash, Claude Sonnet 5, GPT-5.5 Thinking이며, 동일한 업무 과업 프롬프트를 입력한 뒤 업무 맥락 이해도, 정확성, 한국어 자연스러움, 형식 준수, 환각 방지, 실무 활용성을 기준으로 평가하였다.

평가 결과, Gemini 3.5 Flash는 교육 운영 업무 맥락을 구체적으로 표현하고 자연스러운 문장을 생성하는 강점이 있었다. Claude Sonnet 5는 정중하고 균형 잡힌 회신 메일을 작성하는 데 강점을 보였다. GPT-5.5 Thinking은 상대적으로 간결한 표현을 사용했지만, 확인되지 않은 정보를 임의로 생성하지 않고 본문과 확인 필요 사항을 분리하는 능력이 가장 안정적이었다.

따라서 본 과제의 최종 선정 모델은 **GPT-5.5 Thinking**이다. GPT-5.5 Thinking은 교육 운영 회사의 신입사원 및 사회초년생이 실제 업무에서 잘못된 일정, 내부 규정, 담당자 정보, 개인정보를 임의로 포함하지 않도록 돕는 데 가장 적합했다. 또한 메일 제목, 메일 본문, 확인 필요 사항, 수정 제안의 출력 형식을 안정적으로 유지하여 재사용 가능한 업무 자동화 프롬프트 시스템의 기준 모델로 적절하다고 판단하였다.

실행 로그 검증에서도 최종 프롬프트는 10턴 이상의 대화 동안 과정명, 기존 일정, 수신자, 작성자 역할, 금지 사항을 대체로 안정적으로 유지하였다. 중간에 개인정보 처리 방식, 회사명 미기재, 서명 문구, 답변 예상 시점 미기재 등 조건이 변경되었을 때도 변경된 부분만 반영하고 기존 요구사항은 유지하였다. 최종적으로 실행 로그의 문맥 유지 검증 결과는 Pass로 판정하였다.

### 5-2. 배운 점

본 과제를 통해 LLM을 업무에 활용할 때 단순히 “좋은 문장”을 생성하는 것보다, **업무 목적에 맞는 기준을 세우고 검증하는 과정**이 중요하다는 점을 확인하였다.

첫째, LLM은 확률적 생성 방식으로 답변을 만들기 때문에, 사용자가 명확히 제공하지 않은 정보도 그럴듯하게 보완하려는 경향이 있다. 예를 들어 변경 가능한 날짜, 내부 규정, 회신 예상 시점, 담당자 정보는 실제 근거가 있어야 하는 정보인데, 프롬프트 제약이 약하면 모델이 예시처럼 제시하거나 본문에 포함할 가능성이 있다. 따라서 업무 자동화에서는 “모르면 모른다고 말하기”, “확인 필요 사항으로 분리하기”, “임의 생성 금지” 같은 안전장치가 필요하다.

둘째, Zero-shot, Few-shot, 단계적 추론 유도 프롬프트의 차이를 실제 과업에 적용해볼 수 있었다. 단순히 “메일을 작성해줘”라고 지시하는 v1 프롬프트는 빠르게 결과를 얻을 수 있지만, 정보 부족 상황이나 환각 방지에는 취약했다. 반면 v2 프롬프트는 먼저 목적, 수신자, 상황, 필수 포함 항목, 금지 사항을 파악하도록 설계했기 때문에 더 안정적인 결과를 만들 수 있었다. Few-shot 예시는 모델이 원하는 출력 형식과 대응 방식을 학습하는 데 도움을 주었다.

셋째, LLM 비교는 느낌이 아니라 평가 축과 근거를 통해 진행해야 한다는 점을 확인하였다. Gemini 3.5 Flash는 문장이 자연스럽고 풍부했지만, 확인되지 않은 운영 요소를 적극적으로 제안하는 경향이 있었다. Claude Sonnet 5는 균형 잡힌 결과를 제공했지만, 일부 내부 기준 관련 표현은 추가 확인이 필요했다. GPT-5.5 Thinking은 문장이 다소 간결했지만, 환각 방지와 형식 준수에서 가장 안정적이었다. 이처럼 모델마다 강점이 다르기 때문에 업무 목적에 맞는 평가 기준을 먼저 정하는 것이 중요하다.

넷째, 페르소나와 시스템 프롬프트는 결과 품질에 직접적인 영향을 준다. “업무용 메일 작성 AI”라고만 지시했을 때보다, “교육 운영 업무에 특화된 비즈니스 메일 작성 코치”로 역할을 구체화했을 때 출력 결과가 더 일관되었다. 또한 정확성, 개인정보 보호, 형식 준수, 자연스러운 한국어, 친절함의 우선순위를 명시함으로써 모델이 어떤 기준을 우선해야 하는지 더 분명해졌다.

다섯째, 실행 로그를 통해 프롬프트 개선은 한 번에 끝나는 작업이 아니라 반복적으로 수정해야 하는 과정임을 확인하였다. 실제 대화 중 사용자가 개인정보 항목을 직접 넣어달라고 요청했을 때, AI가 이를 본문에 그대로 반영하는 문제가 발생했다. 이후 “필요한 경우 별도 안내” 방식으로 수정하면서 개인정보 보호 원칙을 더 명확히 반영할 수 있었다.

### 5-3. 한계

본 과제에는 다음과 같은 한계가 있다.

첫째, 모델 비교가 하나의 대표 시나리오를 중심으로 진행되었다. 수강 일정 변경 문의 회신 메일은 교육 운영 업무에서 자주 발생하는 상황이지만, 모든 업무용 메일 유형을 대표한다고 보기는 어렵다. 자료 요청, 환불 문의, 수료 안내, 민원 응대, 외부 강사 협조 요청 등 다른 상황에서도 추가 테스트가 필요하다.

둘째, 각 모델의 응답은 웹 환경에서 생성되었기 때문에 temperature, top_p, max_tokens 등 세부 설정값을 직접 통제하지 못했다. 따라서 동일 모델이라도 사용 시점이나 서비스 환경에 따라 결과가 달라질 수 있다. 특히 무료 버전 모델은 응답 횟수나 모델 선택에 제한이 있을 수 있으므로, 완전히 동일한 조건의 실험이라고 보기는 어렵다.

셋째, 환각 검증 질문은 실제 회사의 내부 규정이나 정책 문서를 기반으로 한 것이 아니라 가상의 교육 운영 상황을 기준으로 설계되었다. 따라서 실제 기업 환경에 적용하려면 회사의 개인정보 처리 기준, 환불 규정, 교육 일정 변경 기준, 고객 응대 가이드라인과 연결한 검증이 추가로 필요하다.

넷째, 실행 로그는 10턴 이상의 대화로 문맥 유지 여부를 검증했지만, 더 긴 대화나 여러 업무가 동시에 섞인 상황까지 충분히 검증하지는 못했다. 실제 업무에서는 수강생 정보, 내부 일정표, 강사 일정, 시스템 접수번호 등 다양한 정보가 동시에 등장할 수 있으므로, 복합적인 조건에서도 문맥을 유지하는지 추가 검증이 필요하다.

다섯째, 최종 프롬프트는 메일 초안 작성에는 적합하지만, 실제 발송 승인, 개인정보 저장, 내부 시스템 연동까지 자동화하는 수준은 아니다. 따라서 현재 단계의 시스템은 “메일 작성 보조 도구”로 보는 것이 적절하며, 실제 발송 전에는 반드시 사람이 최종 검토해야 한다.

### 5-4. 추후 개선 방향

추후에는 본 프롬프트 패키지를 다음 방향으로 개선할 수 있다.

첫째, 업무 시나리오를 확장할 필요가 있다. 현재는 수강 일정 변경 문의 회신 메일을 중심으로 검증했지만, 향후에는 교육 안내 메일, 자료 요청 메일, 수료 안내 메일, 환불 문의 회신 메일, 외부 강사 협조 요청 메일 등 다양한 상황별 템플릿을 추가할 수 있다.

둘째, 회사 내부 규정과 연결한 검증 체계를 만들 수 있다. 예를 들어 일정 변경 가능 기준, 환불 규정, 개인정보 요청 가능 항목, 회신 기한 등이 문서화되어 있다면, LLM이 해당 문서를 근거로 답변하도록 설계할 수 있다. 이 경우 “확인 필요”로 분리하는 수준을 넘어, 실제 근거 기반 메일 작성이 가능해진다.

셋째, 개인정보 보호 규칙을 더 강화할 수 있다. 실행 로그에서 확인된 것처럼 사용자가 개인정보 항목을 직접 넣어달라고 요청하면 AI가 이를 본문에 반영할 수 있다. 따라서 최종 프롬프트에는 “개인정보 항목은 사용자가 요청하더라도 필요성과 내부 기준을 먼저 확인하고, 기본적으로 직접 나열하지 않는다”는 규칙을 추가하는 것이 좋다.

넷째, 출력 결과를 검토하는 체크리스트를 추가할 수 있다. 예를 들어 최종 메일 하단에 “발송 전 점검” 항목을 만들어 과정명, 일정, 수신자, 개인정보, 내부 규정, 회신 시점을 확인하도록 하면 신입사원이 실수를 줄일 수 있다.

다섯째, 커스텀 인스트럭션 또는 GPTs/에이전트 기능을 활용해 재사용 가능한 봇으로 배포할 수 있다. 이 경우 사용자는 매번 전체 프롬프트를 입력하지 않아도, 정해진 입력 템플릿에 필요한 정보만 채워 넣어 교육 운영 메일 초안을 생성할 수 있다.

여섯째, 멀티모달 확장도 가능하다. 예를 들어 교육 일정표, 신청자 목록, 강의 자료 안내문을 이미지나 문서로 제공하면, LLM이 해당 자료를 바탕으로 안내 메일 초안을 작성하도록 확장할 수 있다. 다만 이 경우 첨부파일 내용을 잘못 해석하거나 없는 내용을 생성하지 않도록 출처 확인 및 검증 절차를 함께 설계해야 한다.

### 5-5. 최종 요약

본 과제는 교육 운영 회사의 신입사원 및 사회초년생이 업무용 메일을 안정적으로 작성할 수 있도록 돕는 LLM 기반 업무 자동화 프롬프트 패키지를 설계한 것이다.

3종의 LLM을 비교한 결과, 문장 자연스러움에서는 Gemini 3.5 Flash와 Claude Sonnet 5가 강점을 보였으나, 환각 방지, 형식 준수, 확인 필요 사항 분리, 실무 적용 안정성 측면에서는 GPT-5.5 Thinking이 가장 적합했다. 이에 따라 최종 모델은 GPT-5.5 Thinking으로 선정하였다.

최종 프롬프트는 교육 운영 업무에 특화된 페르소나, 출력 형식 규칙, 개인정보 보호 원칙, 환각 방지 규칙, 확인 질문 기준, Few-shot 예시, 환각 검증 설계를 포함한다. 특히 실행 로그에서 확인된 개인정보 직접 요청 문제를 반영하여, 개인정보 항목은 기본적으로 직접 나열하지 않고 필요한 경우 별도 안내하도록 보완하였다. 실행 로그 검증 결과, 최종 프롬프트는 10턴 이상의 대화에서도 핵심 문맥을 유지했으며, 조건 변경이 발생했을 때 변경된 부분만 반영하고 기존 요구사항은 유지하였다.

따라서 본 프롬프트 패키지는 교육 운영 업무에서 반복적으로 발생하는 메일 초안 작성 업무를 보조하는 데 활용 가능하며, 실제 적용 시에는 내부 규정과 개인정보 처리 기준을 연결하여 추가 검증하면 더 안정적인 업무 자동화 도구로 발전시킬 수 있다.
